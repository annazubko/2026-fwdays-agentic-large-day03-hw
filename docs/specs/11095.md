# Spec: Keyboard Shortcut for Cycling Stroke Style (Issue #11095)

---

## 1. Context / Problem

`actionChangeStrokeStyle` in `packages/excalidraw/actions/actionProperties.tsx` has no `keyTest` — the stroke style property (`solid`, `dashed`, `dotted`) is only reachable via the properties panel or the Command Palette. Both require a mouse interaction that interrupts keyboard-driven drawing workflows.

Users who alternate frequently between stroke styles (e.g. solid for confirmed nodes, dotted for tentative ones in diagram sessions) have no way to stay on the keyboard throughout.

**Key conflict:** The issue proposes bare `9`, but that key is already bound to the Image tool via `findShapeByKey` in `packages/excalidraw/components/App.tsx:5025`. Remapping it would break an existing feature. The chosen key is `Shift+9` — currently unbound, layout-independent (keyed on `event.code === CODES.NINE`), and consistent with the existing zoom shortcuts `Shift+1` / `Shift+2` / `Shift+3` defined in `actionCanvas.tsx:413-418`.

---

## 2. Goal

Allow users to cycle through the three stroke styles (`solid → dashed → dotted → solid`) using a single keyboard shortcut (`Shift+9`) without touching the mouse or opening any panel.

---

## 3. Scope

- Add `keyTest` to `actionChangeStrokeStyle` that matches `Shift+9` (and only `Shift+9` — no modifier combinations).
- Update `perform` to compute the next style in the cycle when called without an explicit value (keyboard path); the existing panel call that passes an explicit value must continue to work unchanged.
- Introduce `getNextStrokeStyle(current: StrokeStyle): StrokeStyle` as a pure helper function to centralize and isolate cycle logic.
- Register `"changeStrokeStyle"` in `shortcuts.ts` so the shortcut is automatically rendered in the Help dialog (`?`) and in stroke style control tooltips.
- Apply the new style to the current selection when elements are selected, or to `appState.currentItemStrokeStyle` when there is no selection.
- Cover the new behavior with unit tests in `actionProperties.test.tsx`.

---

## 4. Out of Scope

- Bare `9` key — reserved for the Image tool; must not be changed.
- Existing shortcuts `1`–`8`, `0` — must not be affected.
- Stroke style UI redesign.
- User-customizable shortcut mapping.
- New stroke styles beyond `solid`, `dashed`, `dotted`.
- Changes to stroke width, fill style, or color.
- Chord mode (`Shift+9` then `1`/`2`/`3` for direct selection) — deferred to a follow-up issue.
- Modification of any protected file: `scene/renderer.ts`, `data/restore.ts`, `actions/manager.ts`, `types.ts`.
- Adding new npm packages.

---

## 5. Requirements

### R1 — Shortcut activation

`Shift+9` fires `actionChangeStrokeStyle` when ALL of the following are true:

- `event.shiftKey === true`
- `event.code === CODES.NINE`
- `event.altKey === false`
- `event[KEYS.CTRL_OR_CMD] === false`
- Canvas has focus (not a text editor, `<input>`, or `<textarea>`)

### R2 — Cycle order

The cycle order is fixed and wraps:

```
solid → dashed → dotted → solid → …
```

The next style is computed from `appState.currentItemStrokeStyle`, not from the selection, ensuring deterministic behavior for all cases including mixed selections.

### R3 — Selection behavior

- **Elements selected:** apply `nextStyle` to every selected element via `changeProperty`. The `strokeStyle` property is written to all selected elements; elements whose renderers do not use `strokeStyle` (e.g. images, frames) are unaffected visually. No per-type filtering is added — this matches the existing behavior of all other property actions.
- **No elements selected:** update `appState.currentItemStrokeStyle` to `nextStyle`. All subsequently drawn elements will use `nextStyle`.

### R4 — Visual feedback

The stroke style control in the properties panel and floating toolbar must reflect `nextStyle` immediately after `Shift+9` fires. No additional user action (click, focus, re-render trigger) is required.

### R5 — Discoverability

`Shift+9` must appear:

- In the Help dialog (opened with `?`) next to the stroke style label.
- In the tooltip of the stroke style control in the properties panel.

Both are satisfied automatically by registering the shortcut in `shortcuts.ts`.

### R6 — Panel call unchanged

Calling `actionChangeStrokeStyle.perform(elements, appState, explicitValue)` with a non-null `explicitValue` must produce the same result as before this change. The keyboard path (`value === undefined`) must not affect it.

## 7. Technical Approach

### 7.1 `actionProperties.tsx`

- Import `CODES` from `@excalidraw/common` (already exported, used in `actionCanvas.tsx`).
- Add `keyTest` to `actionChangeStrokeStyle`: match `Shift+9` using `event.code === CODES.NINE` with `shiftKey` guard, reject `altKey` and `Ctrl/Cmd`.
- Extract a pure helper `getNextStrokeStyle(current)` that returns the next value in the `solid → dashed → dotted` cycle. Place it at module scope, outside the action object.
- Update `perform`: when `value` is `undefined` (keyboard path), call `getNextStrokeStyle(appState.currentItemStrokeStyle)`. When `value` is provided (panel path), use it directly — no behavior change for existing callers.

### 7.2 `shortcuts.ts`

- Add `"changeStrokeStyle"` to the `ShortcutName` union type.
- Add `changeStrokeStyle: [getShortcutKey("Shift+9")]` to `shortcutMap`. This automatically surfaces the shortcut in the Help dialog and tooltips.

### 7.3 `actionProperties.test.tsx`

- New `describe("actionChangeStrokeStyle")` block.
- Test `keyTest`: true for `Shift+9`, false for bare `9`, `Ctrl+Shift+9`, `Alt+Shift+9`, unrelated keys.
- Test `perform` cycle: `solid → dashed`, `dashed → dotted`, `dotted → solid`.
- Test `perform` direct value: explicit value passed → used as-is (panel backward compatibility).

## 9. Risks / Edge Cases

| Scenario | Risk | Mitigation |
| --- | --- | --- |
| Bare `9` regression | Image tool stops working | `keyTest` guards on `event.shiftKey === true`; bare `9` never matches |
| Mixed selection (compatible + incompatible elements) | Property written to all; visually no effect on images/frames | Matches existing `changeProperty` behavior — no extra filtering needed; verify rectangle changes, image unchanged |
| Mixed selection with different current stroke styles | Non-deterministic next style | Cycle always reads from `appState.currentItemStrokeStyle`, not from selection — documented in R2 |
| `Shift+9` inside text editor | Inserts `(` or triggers unintended change | `ActionManager` gates `keyTest` dispatch behind focus checks; verify with manual test AC7 |
| `Shift+9` in browser address bar / DevTools | Unexpected browser behavior | Out of scope — standard browser shortcut guard applies |
| `CODES.NINE` not exported from `@excalidraw/common` | TypeScript error | Verify `CODES` export before implementation; it is already used in `actionCanvas.tsx` |
